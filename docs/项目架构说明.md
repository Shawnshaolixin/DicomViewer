# 项目架构说明

## 1. 架构目标

这个项目是一个学习型 WPF 医学影像查看器，当前架构目标不是一步做全，而是先把“分层边界”和“主链路”搭清楚，便于后续持续扩展。

当前设计强调三点：

1. UI 不直接处理 DICOM 解析细节。
2. 业务状态由 Application 层统一编排。
3. 外部库能力集中在 Infrastructure 层封装。

## 2. 解决方案结构

当前解决方案包含以下项目：

1. src/DicomViewer.Wpf
2. src/DicomViewer.Application
3. src/DicomViewer.Domain
4. src/DicomViewer.Infrastructure
5. src/DicomViewer.Rendering
6. src/DicomViewer.Shared
7. tests/DicomViewer.Tests

## 3. 各项目职责

### 3.1 DicomViewer.Wpf

负责 WPF 界面、Prism 绑定、命令交互和视图状态展示。

当前主要内容：

1. App.xaml.cs：Prism 容器注册、Shell 创建和 ViewModelLocator 配置
2. MainWindow.xaml：主窗口布局和 Prism 自动绑定入口
3. MainViewModel.cs：基于 BindableBase / DelegateCommand 的界面命令、快照应用、BitmapSource 创建

### 3.2 DicomViewer.Application

负责用例编排和工作区状态组织，是 UI 和底层实现之间的协调层。

当前主要内容：

1. IStudyCatalogService：抽象数据来源
2. IViewportImageService：抽象图像位图来源
3. WorkspaceService：统一管理当前序列、切片、工具、窗宽窗位、缩放等状态
4. WorkspaceSnapshot：把界面所需数据一次性打包回传给 WPF

### 3.3 DicomViewer.Domain

负责表达医学影像查看器中的核心业务对象和值对象。

当前主要内容：

1. Patient、Study、Series、ImageInstance
2. WindowLevel、PixelSpacing、ViewTransform
3. ViewerToolMode、ModalityType

这一层不依赖 WPF 和 fo-dicom，便于后续保持核心模型稳定。

### 3.4 DicomViewer.Infrastructure

负责与外部文件、第三方库和真实数据源打交道。

当前主要内容：

1. FileSystemStudyCatalogService：扫描目录并读取 DICOM 元数据
2. DicomViewportImageService：读取单帧灰度像素并转换为 Gray8 缓冲

这一层是当前项目里和 fo-dicom 耦合最深的部分。

### 3.5 DicomViewer.Rendering

负责构建视口的描述信息。

当前主要内容：

1. RenderRequest：封装当前视口渲染输入
2. RenderedViewport：封装视口标题、状态和占位文本
3. PlaceholderRenderService：生成当前版本的说明性渲染结果

说明：
当前真实像素位图并不在这个项目中生成，而是在 Infrastructure 中提供。以后如果项目继续演进，可以把这部分能力逐步回收到 Rendering 中。

### 3.6 DicomViewer.Shared

负责预留的通用基础设施项目。

当前说明：

1. WPF 层已切换到 Prism 的 BindableBase 和 DelegateCommand
2. Shared 项目当前保留，便于后续继续沉淀跨层公共代码

### 3.7 DicomViewer.Tests

负责基本回归验证，确保主要学习链路不被后续修改破坏。

当前覆盖点：

1. 样例/空目录导入
2. 单帧灰度图像读取
3. 窗宽窗位影响像素映射
4. 多帧图像切换
5. 工作区测量标注状态

## 4. 当前核心调用链

### 4.1 导入链路

1. MainWindow.xaml 通过 Prism ViewModelLocator 自动绑定到 MainViewModel
2. MainViewModel 调用 WorkspaceService.LoadAsync
3. WorkspaceService 调用 IStudyCatalogService
4. FileSystemStudyCatalogService 扫描目录、读取 DICOM 元数据并组装 Domain 模型
5. WorkspaceService 生成 WorkspaceSnapshot
6. MainViewModel 把快照内容同步到界面属性

### 4.2 图像显示链路

1. 用户切换序列或切片
2. MainViewModel 调用 WorkspaceService.SelectSeries 或 MoveSlice
3. WorkspaceService 选出当前 ImageInstance
4. WorkspaceService 调用 IViewportImageService
5. DicomViewportImageService 读取像素并输出 Gray8 缓冲
6. MainViewModel 把 ViewportImageData 转为 BitmapSource
7. MainWindow.xaml 中的 Image + Canvas 叠加层显示图像与测量标注

### 4.3 窗宽窗位链路

1. 用户点击预设或 WW/WL 调整按钮
2. MainViewModel 调用 WorkspaceService.ApplyWindowLevelPreset 或 AdjustWindowLevel
3. WorkspaceService 更新当前有效 WindowLevel
4. BuildSnapshot 时再次调用 IViewportImageService
5. DicomViewportImageService 使用新的 WindowLevel 重新映射像素
6. 中央视口得到新的 Gray8 图像

### 4.4 交互与标注链路

1. 用户在视口中滚轮缩放、拖拽平移或拖拽调窗
2. MainWindow.xaml.cs 把鼠标事件转发给 MainViewModel
3. MainViewModel 调用 WorkspaceService 更新 ViewTransform、WindowLevel 或测量草稿
4. WorkspaceService 把变更后的图像状态和测量集合写入 WorkspaceSnapshot
5. MainWindow.xaml 使用变换和叠加层重新显示图像与标注

## 5. 当前架构特点

### 5.1 优点

1. 层次清楚，学习成本低。
2. DICOM 解析和 UI 绑定分离。
3. 主要状态都集中在 WorkspaceService，便于追踪。
4. 可以逐步替换实现，不需要推翻现有骨架。

### 5.2 当前不足

1. Prism 容器已经接管 WPF 启动，但模块拆分和区域化导航尚未展开。
2. Rendering 和 Infrastructure 的图像职责边界还不完全理想。
3. 多视口、交互矩阵、测量系统尚未形成独立模块。
4. 当前 ViewModel 仍然偏集中，后续可以继续拆分。

## 6. 后续推荐演进方向

1. 增加自定义视口控件，把缩放、平移、鼠标交互移出 MainWindow.xaml。
2. 把真实位图渲染职责逐步收敛到 Rendering 项目。
3. 为测量与标注单独建立模块。
4. 引入配置、日志、缓存服务。
5. 后续如果模块继续增多，可继续沿 Prism 方向补充模块注册、区域导航和事件聚合。
