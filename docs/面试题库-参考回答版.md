# 面试题库（参考回答版）

## 1. 使用方式

这份文档基于 `docs/面试题库-面试版.md` 扩展而来，目标不是提供“标准答案”，而是提供一版适合面试表达的参考回答骨架。

建议使用方式：

1. 先看题目和回答要点，确认自己是否真正理解。
2. 再看这里的参考回答，学习如何把知识点组织成面试表达。
3. 最后结合本项目中的实现细节，替换成你自己的表述。

## 2. DICOM 基础

### 2.1 DICOM 的 Patient / Study / Series / Instance 分层是什么？

参考回答示例：

“我理解 DICOM 的核心组织方式是 Patient、Study、Series、Instance 四层。Patient 是患者，Study 对应一次检查，Series 是检查下的一组序列，Instance 是具体的图像或对象实例。这个层级在查看器、存储、查询和回取里都很重要。我在这个项目里做本地目录导入时，就是按这个结构把数据整理出来，再提供给界面展示。”

### 2.2 Study Instance UID、Series Instance UID、SOP Instance UID 分别有什么作用？

参考回答示例：

“这几个 UID 是不同层级对象的唯一标识。Study Instance UID 标识一次检查，Series Instance UID 标识检查中的一个序列，SOP Instance UID 标识单个实例。它们的价值不只是唯一性，还体现在分组、关联、发送和追踪上。比如一个系列里的所有图像要归到一起，本质上就要依赖 Series UID 这类标识。”

### 2.3 Window Width / Window Level 是什么？

参考回答示例：

“我会把窗宽窗位理解为显示映射参数，而不是图像原始数据本身。WL 决定显示中心，WW 决定显示范围。对于医学图像来说，不同组织关注的灰度范围不同，所以需要通过 WW/WL 调整把关心的结构显示出来。在这个项目里，我也实现了窗宽窗位预设和手动调整，本质就是重新做像素映射。”

### 2.4 单帧、多帧、灰度、RGB 图像处理上有什么差异？

参考回答示例：

“单帧和多帧最大的差异在于浏览模型不同，多帧要考虑当前帧切换。灰度图通常重点在像素值映射和窗宽窗位，RGB 则要关注多通道数据组织。对查看器来说，这不只是解码差异，还会影响缩略图、当前帧展示、失败回退等逻辑。这个项目里我既处理了灰度显示，也补了基础 RGB 支持和多帧浏览。”

### 2.5 DICOM 文件和 DICOM 网络服务是什么关系？

参考回答示例：

“我理解 DICOM 文件更偏本地存储和交换，DICOM 网络服务更偏系统间通信。它们底层都围绕 DICOM 数据集，但一个解决‘怎么保存和读取’，一个解决‘怎么查询、发送、回取和反馈状态’。所以做一个医学影像项目时，如果只会读文件，其实还不够，还要理解 PACS、MWL、MPPS 这些网络侧能力。”

## 3. PACS / 网络通信

### 3.1 C-ECHO、C-STORE、C-FIND、C-MOVE 分别做什么？

参考回答示例：

“C-ECHO 我会理解成连通性检查，类似确认双方能不能正常通信。C-STORE 是发送影像对象，C-FIND 是查询，C-MOVE 是请求远端把对象发到指定接收端。它们一起构成了最基础的 PACS 交互闭环。我在这个项目里也做了从连通性验证、发送到查询和回取的最小闭环。”

### 3.2 AE Title、Host、Port 分别代表什么？

参考回答示例：

“AE Title 是 DICOM 应用实体的标识，Host 是网络地址，Port 是监听端口。做联调时这三者都要匹配，不然很容易出现连不上、被拒绝或者发错目标的问题。我的理解是，AE Title 更偏 DICOM 语义层面的身份，Host 和 Port 更偏网络层面的定位。”

### 3.3 为什么做回取时要考虑本地接收端？

参考回答示例：

“因为 C-MOVE 并不是把数据直接回到当前查询连接，而是让远端把结果推送到指定的接收端。所以本地如果没有可用的 Store SCP，回取链路其实是不完整的。这个点很容易在初学 PACS 时忽略，但一旦做真正的 Query/Retrieve，就必须考虑本地接收能力。”

### 3.4 PACS 发送失败时要考虑哪些设计？

参考回答示例：

“我会重点考虑四件事：第一，要给用户明确反馈，不要只显示失败；第二，要记录发送结果和错误原因；第三，要区分临时网络失败和业务级错误，决定是否允许重试；第四，要避免本地会话状态和远端发送状态完全脱节。医疗软件里失败不是简单弹个框就结束，后面通常还要能追踪和补救。”

## 4. 医疗工作流

### 4.1 MWL 的作用是什么？

参考回答示例：

“MWL 的核心价值是把待执行检查任务从上游系统带到设备端，让设备侧不需要反复手工录入患者和检查信息。这样既提高效率，也减少信息不一致的风险。对我来说，MWL 不是单纯的‘查列表’，而是把设备工作纳入标准化工作流的入口。”

### 4.2 MPPS 的作用是什么？

参考回答示例：

“MPPS 主要是把检查执行状态反馈回去，比如开始、完成或者中止。它关注的是过程状态，而不是影像文件本身。我的理解是，如果说 MWL 解决的是‘接到什么任务’，那 MPPS 解决的就是‘这个任务现在执行到什么程度了’。”

### 4.3 一个检查从排队到完成，通常有哪些关键状态？

参考回答示例：

“我一般会把它拆成待执行、已选择或已建会话、联锁通过、曝光中、处理中、已完成或已中止几个阶段。这样做的好处是业务动作和系统状态是一一对应的，后续不管做审计、错误处理还是界面反馈，都会更清晰。这个项目里的曝光控制台其实就是沿着这个思路做状态流转的。”

### 4.4 为什么医疗流程里要保留审计日志？

参考回答示例：

“因为医疗软件比普通业务软件更强调可追踪性。审计日志不仅是为了排错，还关系到后续回溯‘谁在什么时间做了什么操作、系统做了什么动作、为什么失败’。尤其是涉及检查任务、发送结果和状态切换时，没有审计日志，很多问题就无法定位。”

### 4.5 本地会话状态和远端工作流状态不一致时怎么办？

参考回答示例：

“我会先把本地状态和远端状态的映射规则定义清楚，然后记录最近一次发送结果和错误原因。之后再根据业务决定是阻断、重试还是进入待补发。关键点是不要把状态只放在内存里，而是要持久化下来，不然一重启就不知道系统到底处在什么状态了。”

## 5. WPF / 客户端开发

### 5.1 MVVM 的主要价值是什么？

参考回答示例：

“MVVM 对我来说最重要的价值是解耦。View 负责展示，ViewModel 负责状态和命令，业务逻辑再往下沉到服务层。这样做之后，界面变化不会直接牵动底层逻辑，测试也更容易写。这个项目里我从传统 WPF 开发继续往前走，就是希望把交互和业务编排真正分开。”

### 5.2 Prism 在这个项目里解决了什么？

参考回答示例：

“Prism 帮我解决了容器注册、依赖注入、ViewModel 绑定和页面导航这些基础组织问题。这样我不用在窗口代码里手工拼装太多对象，可以把注意力更多放在业务编排和页面结构上。对于一个逐步扩展的学习项目来说，这种框架化组织方式很有帮助。”

### 5.3 为什么不能把 DICOM 解析直接写进 ViewModel？

参考回答示例：

“如果把 DICOM 解析直接写进 ViewModel，UI 层就会直接耦合 fo-dicom 这类底层库，后面很难测试，也很难替换实现。更重要的是，界面职责会变得很混乱。我的做法是把这些能力放到 Infrastructure，再由 Application 层统一编排，这样结构更稳定。”

### 5.4 图像缩放、平移、测量为什么都和坐标变换有关？

参考回答示例：

“因为用户在界面上操作的是视口坐标，而真正计算测量和图像定位时需要回到图像空间。只要有缩放和平移，界面坐标和图像坐标之间就不再是简单的一一对应。也就是说，图像交互看起来像 UI 问题，本质上其实是坐标映射和状态管理问题。”

### 5.5 大图像或多序列场景下如何考虑性能？

参考回答示例：

“我会从几个角度考虑：减少不必要重绘、把元数据加载和像素加载分开、引入异步和缓存、必要时做缩略图或延迟加载。一个重要原则是不要把重任务都压在 UI 线程上。对查看器来说，性能问题很多时候不是单点算法问题，而是整体加载策略问题。”

## 6. 架构 / 工程能力

### 6.1 为什么 DICOM 解析更适合放在 Infrastructure 层？

参考回答示例：

“因为它本质上依赖外部库和外部数据源，属于具体实现能力，而不是核心业务模型本身。放在 Infrastructure 层后，UI 和 Application 就不会直接依赖这些细节。这样后续无论是换实现、补测试还是扩展其他数据源，代价都会更低。”

### 6.2 Application 层在这个项目里主要承担什么职责？

参考回答示例：

“我把 Application 层理解为用例编排层。它不直接做底层解析，也不直接负责界面展示，而是负责组织业务动作、推进状态、汇总界面需要的数据。比如查看器里的工作区快照、控制台里的检查流程编排，本质上都适合放在这一层。”

### 6.3 如何设计可测试的业务服务？

参考回答示例：

“核心思路是把外部依赖抽象成接口，把状态推进逻辑集中在服务层，让服务本身不依赖具体界面组件。这样测试时就可以替换文件、数据库、网络这些依赖，专门验证业务行为。这个项目里的目录导入、工作流状态、PACS 结果处理，都是沿这个方向做测试的。”

### 6.4 这个项目里测试为什么重要？

参考回答示例：

“因为这个项目覆盖的不只是界面，还有 DICOM 导入、窗宽窗位映射、工作流状态、PACS 结果处理等关键链路。如果没有自动化测试，后面每次继续学习和重构都容易把主链路改坏。对这种持续迭代的学习项目来说，测试本身也是工程能力的一部分。”

### 6.5 如果继续扩展到 MPR、多视口、缓存，你会先动哪一层？

参考回答示例：

“我不会先从界面层硬加功能，而是先看职责边界是否清晰。通常我会优先补 Application 和 Rendering 的抽象，让状态和渲染模型先稳定下来，再让 Infrastructure 和 WPF 去承接具体实现。这样扩展出来的结构更可维护，也不容易把现有代码搅乱。”

## 7. 医疗软件意识

### 7.1 为什么医疗软件更强调稳定性和可追踪性？

参考回答示例：

“因为它承载的是患者和检查相关信息，系统错误不只是体验问题，还可能影响检查流程和结果追踪。所以相比普通业务系统，医疗软件会更强调可靠性、状态一致性、日志和审计能力。这个项目虽然是学习项目，但我也尽量把这些意识放进去了。”

### 7.2 患者信息和检查信息在日志中如何处理？

参考回答示例：

“我的原则是最小化暴露，只记录必要信息，并区分调试日志、业务日志和审计日志的用途。不是所有字段都应该原样打到日志里，尤其是患者敏感信息。面试里如果谈这一点，我会强调医疗软件开发不能只考虑调试方便，也要考虑信息保护。”

### 7.3 软件异常时应该优先保证什么？

参考回答示例：

“我会优先保证状态一致性和数据可恢复，其次才是交互层面的体验。因为一旦状态混乱，后面的问题会越来越难排查。除此之外，还要保证用户能看到明确反馈，并且系统里留下足够的追踪信息，方便后续定位问题。”

### 7.4 哪些失败适合重试，哪些失败应该阻断？

参考回答示例：

“像临时网络失败、瞬时连接异常，一般适合重试；但像关键联锁失败、关键参数错误、数据不一致这类问题，应该优先阻断流程，而不是盲目继续。我的理解是，重试策略一定要建立在错误分类之上，不能所有失败都一把梭重试。”

## 8. 一段适合串场的总结表达

如果面试官让你开放式介绍，可以这样收束：

“这个项目对我最大的价值，不是单纯学会了怎么显示 DICOM 图像，而是让我从 WPF 客户端开发者的视角，开始系统理解医学影像软件的几个核心问题：数据怎么组织、图像怎么显示、工作流怎么推进、系统怎么和 PACS 通信、状态和审计怎么落地。也正因为做了这个项目，我会更关注医疗软件里的分层、状态一致性、失败追踪和可扩展性，而不是只停留在把功能做出来。”
