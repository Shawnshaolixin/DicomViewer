# 曝光控制台模拟器需求与开发规划

## 1. 文档目的

本文档用于定义当前学习项目的下一阶段目标：在现有 WPF 医学影像查看器基础上，扩展为一个可运行的“曝光控制台模拟器 + 采集流程模拟 + PACS 集成 + 阅片回看”学习型工程。

文档目标包括：

1. 明确项目定位，避免把学习项目误做成真实医疗设备控制系统。
2. 固化后续开发范围、术语、模块边界和阶段目标。
3. 作为后续编码、拆任务、补测试和整理简历项目说明的统一依据。

## 2. 项目定位

本项目不是“真实曝光设备操作系统”，而是“面向 .NET 学习和医疗开发入门的曝光控制台工作流模拟系统”。

项目重点是模拟真实影像设备的软件流程，而不是控制真实硬件高压发生器、球管、探测器或安全电路。

当前阶段应坚持以下边界：

1. 模拟设备工作流，不直接控制真实硬件。
2. 模拟图像采集过程，不实现真实物理曝光算法。
3. 接入标准 DICOM/PACS 工作流，不追求医疗器械合规级实现。
4. 优先建立清晰分层、状态机、日志和通信链路。

## 3. 项目目标

本阶段项目目标如下：

1. 在现有查看器基础上新增曝光控制台业务域。
2. 支持从工作列表选择患者和检查任务。
3. 支持设置曝光参数并进行联锁检查。
4. 支持模拟执行一次曝光并生成图像结果。
5. 支持将模拟结果封装为 DICOM 文件。
6. 支持将 DICOM 图像发送到 PACS。
7. 支持在本地查看器中回看本次采集结果。
8. 支持记录关键操作日志和流程状态。

## 4. 范围定义

### 4.1 第一阶段实现范围

第一阶段只追求打通最小闭环：

1. 工作列表加载
2. 检查任务选择
3. 曝光参数设置
4. 联锁检查
5. 曝光执行模拟
6. 伪图像生成
7. DICOM 文件生成
8. DICOM C-STORE 发送到 Orthanc
9. 使用现有查看器回看采集图像
10. 操作审计日志

### 4.2 第二阶段扩展范围

在第一阶段跑通后，可继续补充以下内容：

1. Orthanc 查询与回取
2. 历史检查列表
3. 检查结果本地持久化
4. 配置文件和通信参数持久化
5. 多张图像采集和系列组织
6. 更真实的图像后处理
7. Worklist 模拟服务增强
8. MPPS 状态上报模拟

### 4.3 当前明确不做的内容

为控制复杂度，当前阶段不做以下内容：

1. 真实设备驱动接入
2. 真实高压控制或探测器 SDK 接入
3. 医疗器械法规级安全控制实现
4. 真实剂量计算与物理建模
5. 自研完整 PACS 替代品
6. MPR、多视口联动等高级阅片能力

## 5. 术语与命名约定

为避免后续命名混乱，统一采用如下术语：

### 5.1 ImagingOrder

表示待执行的检查任务，用于工作列表和技师选择场景。

### 5.2 ExamSession

表示一次实际检查执行过程，用于承载流程状态、曝光参数、联锁检查、采集结果和发送状态。

### 5.3 Study

表示 DICOM / PACS 语义下的影像检查归档单元，用于组织 Series 和 ImageInstance。

### 5.4 建议关系

推荐按如下关系理解：

1. ImagingOrder -> 待办检查任务
2. ExamSession -> 执行中的检查流程
3. Study -> 最终生成并归档的影像数据

也就是说：

1. 与工作流、设备状态、曝光步骤相关的对象优先使用 Exam 命名。
2. 与 DICOM、PACS、StudyInstanceUid、Series 组织相关的对象继续使用 Study 命名。

## 6. 业务流程目标

本项目需要模拟一条完整但简化的影像设备工作流：

1. 用户加载工作列表。
2. 用户选择一个待检查任务。
3. 系统进入检查准备状态并加载默认协议参数。
4. 用户调整曝光参数。
5. 系统执行联锁检查。
6. 联锁通过后允许曝光。
7. 系统模拟生成一张或一组采集图像。
8. 系统对图像做基础处理并封装 DICOM。
9. 系统将图像发送到 PACS。
10. 用户可在查看器中回看本次检查结果。

## 7. 功能需求

### 7.1 工作列表

系统应支持以下能力：

1. 加载模拟工作列表数据。
2. 展示患者、检查部位、协议名称、预约时间、当前状态。
3. 允许选择一条任务作为当前检查。
4. 开始检查后生成对应的 ExamSession。

第一阶段可使用内置假数据，不要求接入真实 RIS 或 MWL。

### 7.2 曝光控制台

系统应提供控制台操作区，用于配置并执行检查流程，至少包括：

1. 当前患者与检查信息展示
2. 当前设备状态展示
3. 曝光参数编辑区
4. 联锁状态展示
5. 曝光执行按钮
6. 发送 PACS 按钮
7. 回看入口

建议先支持以下参数：

1. kV
2. mA
3. ms
4. mAs
5. SID
6. BodyPart
7. Projection
8. AEC 开关

### 7.3 设备状态与联锁

系统应模拟设备状态，并以状态驱动界面行为。

至少应包括以下设备状态：

1. Offline
2. Idle
3. Preparing
4. Ready
5. Exposing
6. Processing
7. Error

至少应包括以下联锁检查项：

1. 是否已选择检查任务
2. 探测器是否连接
3. 球管是否预热完成
4. 防护门是否关闭
5. 参数是否越界
6. PACS 是否可用

联锁失败时应满足：

1. 阻止曝光执行
2. 给出可读失败原因
3. 写入审计日志

### 7.4 采集流程模拟

系统应支持模拟一次完整曝光流程。

最小流程包括：

1. 进入 Ready
2. 点击曝光
3. 状态切换到 Exposing
4. 生成模拟图像
5. 状态切换到 Processing
6. 生成本地结果和预览
7. 返回 Ready 或进入下一步发送状态

第一阶段图像生成方式应保持简单可控：

1. 允许基于模板灰度图或程序生成图案生成图像。
2. 可根据 kV、mAs 对亮度和对比度做简单映射。
3. 可叠加少量噪声，增强“采集结果”感。

### 7.5 DICOM 文件生成

系统应支持把模拟采集结果封装成标准 DICOM 文件。

至少应写入以下信息：

1. Patient 基础信息
2. Study / Series / SOP Instance UID
3. StudyDescription / SeriesDescription
4. Modality
5. Rows / Columns / PixelData
6. WindowWidth / WindowCenter
7. 采集时间
8. 基础曝光参数

第一阶段建议使用 fo-dicom 实现 DICOM 生成。

### 7.6 PACS 集成

系统应支持将生成的 DICOM 图像发送到 PACS。

第一阶段要求：

1. 使用 Orthanc 作为学习阶段 PACS 服务器。
2. 支持 C-STORE 发送。
3. 明确反馈发送成功或失败结果。
4. 记录 AE Title、目标地址、端口等基础配置。

第二阶段再考虑：

1. Query / Retrieve
2. MWL
3. MPPS

当前学习实现已先补两条最小查询与回取闭环：

1. 面向 Orthanc REST 的快速查询与下载，用于快速回看。
2. 面向标准 DICOM 的 C-FIND / C-MOVE 学习链路，用于理解标准查询回取流程。

其中 C-MOVE 依赖本地接收端配置和 Orthanc 可达性，仍以本机或同网段学习环境为主要目标场景。

### 7.7 阅片回看

系统应复用现有查看器作为回看界面，至少支持：

1. 加载本次采集生成的 DICOM 图像
2. 显示 Study / Series / Instance 信息
3. 支持已有窗宽窗位和基础浏览能力
4. 为后续从 PACS 拉取回看预留扩展位

### 7.8 审计日志与配置

系统应记录关键流程节点，并允许后续持久化。

第一阶段至少记录以下事件：

1. 工作列表加载
2. 选择检查任务
3. 修改曝光参数
4. 联锁检查通过/失败
5. 曝光执行开始/完成
6. DICOM 生成
7. PACS 发送开始/成功/失败

第一阶段配置项至少包括：

1. 本机 AE Title
2. PACS AE Title
3. PACS Host
4. PACS Port
5. 本地输出目录
6. 默认参数范围

若进入 Orthanc 查询与回取阶段，建议额外配置 Orthanc REST Port，不与 DICOM 监听端口混用。

若进入标准 C-MOVE 阶段，还应配置本地接收 Host / Port，并确保 Orthanc 能访问该地址。

## 10. 核心状态机建议

### 10.1 ExamSession 状态

建议使用以下状态：

1. Scheduled
2. InProgress
3. Ready
4. Acquiring
5. Processing
6. Sending
7. Completed
8. Failed
9. Cancelled

### 10.2 建议流转

最小流转建议如下：

1. Scheduled -> InProgress
2. InProgress -> Ready
3. Ready -> Acquiring
4. Acquiring -> Processing
5. Processing -> Ready
6. Processing -> Sending
7. Sending -> Completed
8. 任意关键状态 -> Failed

### 10.3 DeviceState 状态

建议使用以下状态：

1. Offline
2. Idle
3. Preparing
4. Ready
5. Exposing
6. Processing
7. Error

## 11. 第一阶段开发顺序

建议按照以下顺序推进：

1. 先补 Domain 的流程对象、状态枚举和值对象。
2. 再实现 Application 的 ExamWorkflowService。
3. 再实现 MockWorklistService 和 MockExposureSimulationService。
4. 再实现 SimulatedDicomBuilder。
5. 再接 OrthancStoreService。
6. 最后接 WPF 控制台界面。

这样做的原因是：

1. 先建立流程骨架，再接外部依赖。
2. 先确保状态流转正确，再做界面细节。
3. 有利于单元测试优先覆盖业务流程。

## 12. 第一阶段验收标准

第一阶段完成时，至少应满足以下标准：

1. 可以加载一组模拟工作列表数据。
2. 可以选择一条任务并创建检查会话。
3. 可以设置基础曝光参数。
4. 可以执行联锁检查并在失败时阻止曝光。
5. 可以生成一张本地 DICOM 图像。
6. 可以把生成的图像发送到 Orthanc。
7. 可以在现有查看器中回看本次采集结果。
8. 可以查看关键操作日志。

## 13. 测试建议

建议至少补充以下测试：

1. 工作列表加载测试
2. 未选择任务时联锁失败测试
3. 参数越界时联锁失败测试
4. 曝光成功后状态流转测试
5. DICOM 文件生成测试
6. PACS 发送结果处理测试
7. 控制台快照组装测试

## 14. 学习重点建议

这个项目最值得学习的重点不是界面，而是以下能力：

1. 医疗影像工作流建模
2. 状态机设计
3. DICOM 基础对象和通信流程
4. 应用层编排与基础设施解耦
5. 审计日志和配置管理意识

建议开发过程中同步补充以下知识：

1. Study / Series / Instance 关系
2. C-STORE、C-FIND、C-MOVE 基本概念
3. Orthanc 的基本部署和调试方式
4. 医疗软件中联锁、日志、状态驱动的设计习惯

## 15. 当前文档与现有项目的关系

当前仓库中的医学影像查看器仍然是基础工程和回看入口。

本文件定义的是它的下一阶段扩展方向：从“查看器骨架”扩展为“曝光控制台模拟器 + 采集流程模拟 + PACS 集成 + 阅片回看”的完整学习工程。

后续开发时应优先遵循本文件的术语、范围边界和阶段目标，以减少反复改名和推倒重来。
