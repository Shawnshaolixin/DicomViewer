# 曝光控制台模拟器需求基线与后续完善规划

## 1. 文档目的

本文档用于重新整理当前学习项目在“曝光控制台模拟器 + 采集流程模拟 + PACS 集成 + 阅片回看”方向上的需求基线。

本次整理的目标不是重复描述最初设想，而是明确以下三件事：

1. 哪些能力已经完成，应该视为当前版本基线。
2. 哪些能力虽然已有实现，但需求定义、异常处理或工程化程度仍需完善。
3. 哪些内容属于后续扩展范围，不应继续和当前版本混在一起管理。

## 2. 项目定位

本项目仍然是学习型工程，不是实际医疗设备控制软件。

项目定位保持不变：

1. 模拟影像设备工作流，不直接控制真实硬件。
2. 模拟图像采集过程，不实现真实物理曝光算法。
3. 接入标准 DICOM 和 PACS 学习链路，但不追求医疗器械合规级实现。
4. 优先训练分层设计、状态机建模、审计日志、持久化和通信流程理解。

## 3. 当前需求基线

以下能力已经完成，后续不应继续作为“第一阶段待开发功能”管理，而应作为当前版本的既有能力。

### 3.1 工作列表与检查会话

系统已具备以下能力：

1. 加载模拟工作列表数据。
2. 展示患者、检查、部位、协议和预约时间等信息。
3. 选择工作列表任务并创建对应检查会话。
4. 使用 ExamSession 承载当前流程状态、参数、结果和发送记录。

### 3.2 曝光控制台

系统已具备以下能力：

1. 提供独立曝光控制台页面。
2. 展示当前患者、检查、设备状态和流程状态。
3. 编辑曝光参数，包括 kV、mA、ms、mAs、SID、BodyPart、Projection 和 AEC。
4. 编辑参数范围并在应用重启后恢复配置。

### 3.3 联锁与状态机

系统已具备以下能力：

1. 模拟探测器连接、球管预热、防护门关闭和 PACS 可用等联锁条件。
2. 执行联锁检查并返回可读失败原因。
3. 在联锁失败时阻止曝光。
4. 驱动 DeviceOperationalState 和 ExamWorkflowStatus 的核心状态流转。

### 3.4 采集模拟与 DICOM 生成

系统已具备以下能力：

1. 模拟一次曝光流程。
2. 生成伪灰度图像结果。
3. 将采集结果封装为可读取的 DICOM 文件。
4. 写入 Patient、Study、Series、SOP Instance、Rows、Columns、PixelData、窗宽窗位和基础曝光参数等信息。

### 3.5 PACS 集成与回看闭环

系统已具备以下能力：

1. 使用 C-ECHO 验证 PACS 连通性。
2. 使用 C-STORE 发送生成的 DICOM 文件到 Orthanc。
3. 基于 Orthanc REST 执行查询和下载，用于快速回看。
4. 基于标准 DICOM 执行 C-FIND 和 C-MOVE 学习链路。
5. 在本地查看器中直接回看本次曝光结果、历史结果和远端回取结果。

### 3.6 审计与持久化

系统已具备以下能力：

1. 记录工作列表加载、参数修改、联锁检查、曝光执行、DICOM 生成、PACS 验证和发送等审计事件。
2. 将 PACS 配置和参数范围持久化到本地 SQLite。
3. 将审计日志持久化到本地 SQLite。
4. 将检查会话状态、最近曝光结果和 PACS 发送记录持久化到本地 SQLite。
5. 在控制台中展示历史检查和最近操作记录。

## 4. 需要完善的需求项

以下内容不是“完全没有实现”，而是当前需求定义不够完整，或者实现已经存在但还没有达到可持续扩展的程度。这些内容应作为下一轮工作的重点。

### 4.1 需求基线与文档同步

当前最先要解决的问题是文档基线落后于实现，后续需求拆分应按以下原则处理：

1. 不再把工作列表、联锁、模拟曝光、DICOM 生成、C-STORE、回看闭环视为第一阶段待开发项。
2. 对已完成能力补充“当前行为定义”，避免后续修改时失去基线。
3. 以“已完成基线 / 待完善项 / 后续扩展”三层结构维护文档。

### 4.2 验收标准细化

当前验收标准过于粗略，需要补充可执行判断条件。

至少应明确：

1. 成功场景的输入、输出和状态变化。
2. 失败场景下的提示、日志和状态回退行为。
3. DICOM 文件生成后的最小可读性要求。
4. PACS 发送、查询、回取成功与失败的判定标准。
5. 回看入口在本地结果、历史结果和远端结果三种路径上的一致行为。

### 4.3 状态机与命名统一

当前流程可运行，但需求层面仍需统一术语和状态描述。

需要完善的点包括：

1. 明确“曝光”“采集”“处理”“发送”在文档、日志和 UI 中的统一用词。
2. 明确 DeviceOperationalState 与 ExamWorkflowStatus 的职责边界。
3. 明确各状态允许的操作、禁止的操作和可见提示。
4. 明确失败后是回到 Ready、InProgress 还是 Failed 的规则。

### 4.4 异常恢复与失败策略

当前最明显的缺口是异常链路定义不足。

需求上至少应补充以下场景：

1. 输出目录不可写或不存在时如何处理。
2. DICOM 生成失败时状态如何回退。
3. PACS 不可达、C-STORE 失败时是否允许重试。
4. Orthanc REST 查询失败时界面如何提示。
5. C-MOVE 回取超时、本地接收端未启动或端口不可达时如何处理。
6. 历史记录中存在失效路径时回看如何降级。

### 4.5 审计日志结构完善

当前已经有日志记录，但需求上尚未明确日志结构规范。

建议至少明确每条审计记录应包含：

1. 时间戳。
2. 事件类型。
3. 检查会话标识。
4. 关联的订单或患者标识。
5. 关键参数摘要。
6. 操作结果和失败原因。
7. 关联的本地文件路径或远端目标信息。

### 4.6 历史检查与结果管理

当前已有历史记录能力，但需求仍偏基础。

下一步应补充：

1. 历史列表筛选条件。
2. 时间排序和分页策略。
3. 历史记录与 PACS 发送记录的关联展示。
4. 历史结果丢失或文件被删除后的降级提示。
5. 是否需要区分“本地生成”“已发送”“已回取”等结果来源。

### 4.7 DICOM 产物约束

当前 DICOM 生成可用，但需求仍需更具体，以支撑后续多图像或多系列扩展。

应进一步明确：

1. Study、Series、SOP Instance UID 的生成和复用策略。
2. SeriesDescription 的命名规则。
3. 单次曝光对应单实例还是可扩展为多实例。
4. 曝光参数写入哪些 DICOM 标签。
5. 回看端最少依赖哪些标签才能稳定加载。

### 4.8 PACS 环境约束与运维前提

当前项目已支持多种查询回取方式，但运行前提尚未在需求中写清楚。

需要补充说明：

1. Orthanc DICOM 端口和 REST 端口应分开配置。
2. 本机 AE Title、远端 AE Title 和本地接收端 AE 配置规则。
3. C-MOVE 依赖本地接收 Host 和 Port 对 Orthanc 可达。
4. 本机防火墙、端口占用和网络可达性对学习环境的影响。
5. 建议的本机或同网段演示环境约束。

### 4.9 非功能需求

当前文档对非功能要求描述不足，后续至少应补以下基线：

1. 控制台常见操作的可接受响应时间。
2. 模拟曝光到生成本地结果的可接受时延。
3. 历史列表和日志列表的规模预期。
4. SQLite 数据增长后的保留与清理策略。
5. 学习数据中患者信息的脱敏边界。

### 4.10 测试与回归范围

当前已有较多测试，但需求文档仍需把测试范围从“建议”提升为“验收组成部分”。

至少应持续覆盖：

1. 工作列表加载。
2. 联锁成功与失败。
3. 参数越界处理。
4. 曝光状态流转。
5. DICOM 文件生成可读性。
6. PACS 发送结果处理。
7. 查询与回取结果处理。
8. 配置、日志、检查会话和发送记录的持久化恢复。

## 5. 当前不作为短期缺口处理的内容

以下内容仍然有价值，但不应和当前待完善项混为一谈。它们属于后续扩展路线。

1. MWL 接入。
2. MPPS 状态上报模拟。
3. 多张图像采集和系列组织增强。
4. 更真实的图像后处理。
5. 独立历史页面、复杂筛选和分页体验优化。
6. 缩略图、标注拖拽编辑和高级阅片能力。
7. 多视口联动、MPR 等高级查看器功能。
8. 真实 RIS、设备 SDK 或硬件驱动接入。

## 6. 推荐的下一轮开发重点

按照当前项目状态，下一轮更合理的优先级如下：

1. 先补需求和文档基线，统一术语、状态和验收标准。
2. 再补异常恢复路径，尤其是 DICOM 生成失败、PACS 不可用和 C-MOVE 接收失败场景。
3. 再完善历史记录和日志结构，使结果管理可持续扩展。
4. 然后细化 DICOM 产物约束和 PACS 运行前提。
5. 最后再继续 MWL、MPPS、多图像采集等扩展功能。

## 7. 建议的验收基线

从当前版本开始，建议把验收拆成“已达成基线”和“待完善基线”两部分。

### 7.1 已达成基线

当前版本应至少视为已满足以下条件：

1. 可以加载模拟工作列表并选择检查任务。
2. 可以编辑曝光参数并执行联锁检查。
3. 可以模拟执行曝光并生成本地 DICOM 文件。
4. 可以执行 C-ECHO 和 C-STORE。
5. 可以基于 Orthanc REST 或标准 DICOM 查询并回取远端检查。
6. 可以在查看器中回看本次曝光结果、历史结果和远端回取结果。
7. 可以持久化配置、审计日志、检查会话和 PACS 发送记录。

### 7.2 待完善基线

下一轮完成时，建议至少满足以下标准：

1. 关键失败路径都有明确提示、日志和状态回退规则。
2. 审计记录结构可追踪单次检查会话的完整闭环。
3. 历史结果管理具备基础筛选和失效降级能力。
4. DICOM 生成约束形成明确规则，便于继续扩展多图像场景。
5. PACS 学习环境的配置前提和限制说明清晰可复现。

## 8. 与现有工程的关系

当前仓库中的查看器、曝光控制台、模拟采集、PACS 通信和回看功能已经形成一个可运行学习工程。

因此，后续开发应基于“已有闭环继续完善”，而不是继续按“从零补第一阶段功能”的思路推进。

本文件今后应承担两个角色：

1. 作为当前版本需求基线文档。
2. 作为下一轮完善和扩展任务的范围边界文档。
