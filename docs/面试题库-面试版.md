# 面试题库（面试版）

## 1. 使用方式

这份题库按 6 个模块整理，每道题只给出“回答要点”，方便先建立答题骨架，再结合项目细节扩展。

## 2. DICOM 基础

### 2.1 DICOM 的 Patient / Study / Series / Instance 分层是什么？

回答要点：

1. Patient 是患者维度。
2. Study 是一次检查。
3. Series 是检查中的一个序列。
4. Instance 是具体图像或对象实例。
5. 查看器和存储组织通常都围绕这层结构展开。

### 2.2 Study Instance UID、Series Instance UID、SOP Instance UID 分别有什么作用？

回答要点：

1. 用于唯一标识不同层级对象。
2. Study UID 标识一次检查。
3. Series UID 标识检查内的一个序列。
4. SOP Instance UID 标识单个对象实例。
5. 在分组、关联、发送和追踪时都很关键。

### 2.3 Window Width / Window Level 是什么？

回答要点：

1. 是像素显示映射参数。
2. WL 决定显示中心，WW 决定显示范围。
3. 不同组织需要不同显示范围。
4. 它改变的是显示效果，不是原始像素本身。

### 2.4 单帧、多帧、灰度、RGB 图像处理上有什么差异？

回答要点：

1. 单帧和多帧的切换逻辑不同。
2. 灰度图通常围绕像素值映射处理。
3. RGB 要考虑多通道数据组织。
4. 查看器还要考虑缩略图、当前帧和失败回退。

### 2.5 DICOM 文件和 DICOM 网络服务是什么关系？

回答要点：

1. 文件解决本地存储和交换。
2. 网络服务解决查询、发送、回取和工作流通信。
3. 两者底层都围绕 DICOM 数据集，但使用场景不同。

## 3. PACS / 网络通信

### 3.1 C-ECHO、C-STORE、C-FIND、C-MOVE 分别做什么？

回答要点：

1. C-ECHO：验证连通性。
2. C-STORE：发送对象。
3. C-FIND：查询对象。
4. C-MOVE：请求远端把对象发送到指定接收端。

### 3.2 AE Title、Host、Port 分别代表什么？

回答要点：

1. AE Title 是 DICOM 应用实体标识。
2. Host 是网络地址。
3. Port 是监听端口。
4. 联调时三者都需要匹配，否则常见问题是连不上或被拒绝。

### 3.3 为什么做回取时要考虑本地接收端？

回答要点：

1. C-MOVE 不是直接把数据回到当前查询连接。
2. 远端通常会把结果推送到指定的 Store SCP。
3. 所以本地必须有可接收实例的服务端能力。

### 3.4 PACS 发送失败时要考虑哪些设计？

回答要点：

1. 明确失败原因和用户提示。
2. 记录发送结果与错误日志。
3. 判断是否允许重试或补发。
4. 避免业务状态和网络状态完全脱节。

## 4. 医疗工作流

### 4.1 MWL 的作用是什么？

回答要点：

1. 把待执行检查任务提供给设备端。
2. 让设备不必手工重复录入关键信息。
3. 帮助统一患者、检查和步骤上下文。

### 4.2 MPPS 的作用是什么？

回答要点：

1. 向外部系统反馈检查执行状态。
2. 常见状态有 Started / In Progress、Completed、Discontinued。
3. 它强调的是过程状态，不是影像本身。

### 4.3 一个检查从排队到完成，通常有哪些关键状态？

回答要点：

1. 待执行。
2. 已选择或已建会话。
3. 联锁通过。
4. 曝光中。
5. 处理中。
6. 已完成或已中止。

### 4.4 为什么医疗流程里要保留审计日志？

回答要点：

1. 便于追踪谁做了什么操作。
2. 便于排查异常和恢复流程。
3. 医疗软件对可追踪性要求更高。

### 4.5 本地会话状态和远端工作流状态不一致时怎么办？

回答要点：

1. 先明确本地状态和远端状态的映射规则。
2. 记录失败原因与最近一次发送结果。
3. 根据业务决定阻断、重试还是待补发。
4. 不要把状态只保存在内存里。

## 5. WPF / 客户端开发

### 5.1 MVVM 的主要价值是什么？

回答要点：

1. 把界面展示和业务逻辑分开。
2. 便于测试和维护。
3. 便于复杂交互下保持结构清晰。

### 5.2 Prism 在这个项目里解决了什么？

回答要点：

1. 帮助做依赖注入和 ViewModel 绑定。
2. 组织页面导航。
3. 减少手工创建对象和界面耦合。

### 5.3 为什么不能把 DICOM 解析直接写进 ViewModel？

回答要点：

1. 会让 UI 层耦合底层库。
2. 不利于测试和替换实现。
3. 会让界面逻辑和基础设施逻辑混在一起。

### 5.4 图像缩放、平移、测量为什么都和坐标变换有关？

回答要点：

1. 用户交互坐标不等于图像原始坐标。
2. 缩放和平移会改变映射关系。
3. 测量要基于稳定的图像空间计算。

### 5.5 大图像或多序列场景下如何考虑性能？

回答要点：

1. 减少不必要重绘。
2. 区分元数据加载和像素加载。
3. 考虑缓存、异步加载和缩略图策略。
4. 不要让 UI 线程直接承担重任务。

## 6. 架构 / 工程能力

### 6.1 为什么 DICOM 解析更适合放在 Infrastructure 层？

回答要点：

1. 它依赖外部库和外部数据源。
2. 属于具体实现，不属于界面或核心领域模型。
3. 放在 Infrastructure 更利于隔离变化点。

### 6.2 Application 层在这个项目里主要承担什么职责？

回答要点：

1. 负责编排用例和状态。
2. 汇总界面需要的快照数据。
3. 连接 UI 和基础设施，不直接承担底层细节。

### 6.3 如何设计可测试的业务服务？

回答要点：

1. 对外部依赖做接口抽象。
2. 把状态推进逻辑放在服务层。
3. 让核心逻辑不依赖具体界面组件。

### 6.4 这个项目里测试为什么重要？

回答要点：

1. 可以验证目录导入、图像读取、窗宽窗位映射。
2. 可以验证工作流状态和 PACS 结果处理。
3. 可以降低后续继续学习和重构时的回归风险。

### 6.5 如果继续扩展到 MPR、多视口、缓存，你会先动哪一层？

回答要点：

1. 先看职责边界，不会直接从 UI 硬加。
2. 多数情况下先补 Application 和 Rendering 的抽象。
3. 再让 Infrastructure 和 WPF 分别承接具体实现与展示。

## 7. 医疗软件意识

### 7.1 为什么医疗软件更强调稳定性和可追踪性？

回答要点：

1. 它承载患者和检查相关信息。
2. 错误可能影响临床流程和结果追踪。
3. 所以比普通信息系统更强调可靠性、日志和审计。

### 7.2 患者信息和检查信息在日志中如何处理？

回答要点：

1. 只记录必要信息。
2. 注意最小化暴露。
3. 区分调试信息、业务日志和审计信息。
4. 避免无控制地输出敏感字段。

### 7.3 软件异常时应该优先保证什么？

回答要点：

1. 状态一致性。
2. 数据不丢失或可恢复。
3. 用户能看到明确反馈。
4. 后续可追踪和排查。

### 7.4 哪些失败适合重试，哪些失败应该阻断？

回答要点：

1. 临时网络失败通常适合重试。
2. 关键联锁失败应阻断流程。
3. 涉及数据不一致、关键参数错误时，不应盲目继续。

## 8. 结合项目讲时的推荐答题主线

如果面试官让你结合项目展开，可以按下面顺序回答：

1. 先讲项目定位：这是一个 WPF 医学影像学习项目。
2. 再讲架构：分层、解耦、Application 编排。
3. 再讲查看器：导入、分组、显示、窗宽窗位、测量。
4. 再讲工作流：工作列表、联锁、曝光、会话状态。
5. 再讲集成：PACS、MWL、MPPS、回看闭环。
6. 最后讲工程化：SQLite、审计、测试、异常处理。

这样既能体现你做过什么，也能体现你理解为什么这么做。
